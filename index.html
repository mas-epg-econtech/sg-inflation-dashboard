1. Data Collection

Prices are scraped daily from three major Singapore supermarket chains via their public-facing APIs and websites:

StoreSourceMethodQuantity Field
NTUC FairPricefairprice.com.sgInternal product API (paginated JSON)metaData.DisplayUnit
Sheng Siongshengsiong.com.sgWebsite product API (JSON)packSize
Cold Storagecoldstorage.com.sgProduct search API (JSON)size

Scraping runs on a daily schedule (8:00 AM SGT) via an automated task. Each run traverses all product categories, extracting: product name, slug, ID, current price, original price (if on promotion), brand, stock status, category hierarchy, and display unit (pack size / quantity label).

Each scrape captures approximately 90,000 unique products across all three stores. Records are deduplicated per (product_slug, source, date) to prevent double-counting from overlapping category listings.

2. CPI Classification

Every scraped product is mapped to a CPI sub-category using the official SingStat 2024 Consumer Price Index classification (157 series at the 4-digit level). Mapping is performed by an LLM (Claude) which reads each product's name, slug, brand, and store category, then assigns the most appropriate CPI series code.

Classification Quality

Of the ~91,700 unique products tracked, 99.9% have been successfully mapped to a valid CPI series. Products that cannot be classified (e.g., gift cards, delivery fees) are assigned 0.00.0.0 and excluded from index calculations.

CPI Basket Weights

Category-level weights are sourced from the SingStat 2024 Household Expenditure Survey weights, which reflect the average Singaporean household's spending pattern. Weights are defined at the 4-digit sub-class level (e.g., "1.01.1.1 Rice" has weight 14, "1.01.2.4 Poultry" has weight 33). These weights determine each category's contribution to the aggregate price index.

3. Price Index Methodology

Chained CPI-Weighted Index

The primary inflation measure. It tracks price changes at the individual SKU level using a chained pairwise approach, then aggregates by CPI category with official SingStat basket weights.

For each date t, find all products present on both t−1 and t.

For each such product j:
  price_relative(j, t) = price(j, t) / price(j, t−1)

For each CPI category i:
  cat_relative(i, t) = mean of price_relative(j, t) for all matched products j in category i

Daily link(t) = [ Σ wi × cat_relative(i, t) ] / [ Σ wi ]

Chained Index(t) = Chained Index(t−1) × Daily link(t),   with Index(day 1) = 100

Because each product is compared only to itself on the previous day, the index is immune to composition effects — products entering or leaving the dataset on other days cannot distort the index. A product only needs to exist on two consecutive days to contribute. The daily link ratios are chained multiplicatively so that the index accumulates over time.

No unit parsing is needed, and every product category can participate. The pairwise basket typically contains ~88,000 SKUs on any given day, covering all CPI categories.

Why this works for all products

Consider a bottle of 120 vitamin tablets at $95.80. We don't need to know the weight per tablet or the total gram count. If the same bottle costs $99.90 the next day, that's a 4.3% price increase, measured cleanly. The product's identity (same product_id and product_slug) guarantees that the quantity is constant by definition — it's the same SKU. This applies equally to cakes sold by the piece, gift hampers, electronics, and any other product where per-gram pricing is meaningless.

Unit Price Chained CPI-Weighted Index (F&B only)

Uses the same chained pairwise methodology as the main index, but instead of comparing shelf prices, it compares price per 100g or per 100ml. This reveals whether apparent price stability masks shrinkflation (same price, smaller pack) or whether price drops reflect genuine per-unit savings.

Scope restriction: This index is computed only for Food & Beverage categories (CPI series 1.01.*), Alcoholic Beverages (1.10.1.*), Personal Care (1.10.3.*), and Medicines & Health Products (1.05.*). Non-grocery categories (furniture, appliances, clothing, electronics) are excluded because their "unit" data is not meaningful for per-gram/per-ml comparison — a washing machine's "kg" is capacity, not weight, and tiny sample sizes in these categories produce volatile, misleading averages.

4. Unit Price Calculation

Each store provides structured quantity information alongside product prices. These are normalised into a common format:

Input: display_unit string (e.g., "2 x 900ml", "1.5kg", "12s x 56g")

Step 1: Parse multiplier and base quantity via regex
   "2 x 900ml" → multiplier=2, qty=900, unit=ml
   "1.5kg" → multiplier=1, qty=1500, unit=g (converted from kg)

Step 2: Compute total in base units
   total = multiplier × qty

Step 3: Compute normalised price
   price_per_100 = (price / total) × 100
   Result: $/100g or $/100ml

Unit conversions applied: kg→g (×1000), L→ml (×1000). Count-based items (e.g., "12 pcs") are tracked as $/piece but excluded from the per-100g/100ml unit price index.

Display Unit Consistency

To prevent discontinuities from inconsistent quantity strings across scrape dates, the best (most recent) display_unit value for each product is propagated to all historical dates. This ensures the same product is measured consistently regardless of when the quantity field was first captured.

5. Promotion Detection

A product is considered "on promotion" when the store API returns both a price (current selling price) and a higher original_price (the regular shelf price). The discount is calculated as:

discount_pct = (original_price - price) / original_price × 100
discount_dollar = original_price - price

Promotion statistics (percentage of catalog on promo, average/median/max discount) are computed per store and across all stores.

7. Limitations & Caveats

  • Online prices only. Scraping captures online listed prices, which may differ from in-store prices. In-store-only promotions and member-exclusive discounts are not captured.
  • Category mapping is approximate. LLM-based classification may occasionally assign products to an incorrect CPI category, especially for ambiguous items (e.g., is coconut milk a dairy product or a beverage?).
  • No quantity weighting. All indices treat all products equally within a category — a $2 packet of rice has the same influence as a $50 premium brand. Official CPI uses expenditure-weighted item sampling.
  • Product churn. Products may be added, removed, or temporarily unavailable across scrape dates. The chained pairwise approach handles this gracefully — a product only needs to exist on two consecutive days — but if a product disappears and reappears at a different price, the gap is invisible to the index.
  • Unit price coverage. The unit price index only covers products with parseable weight/volume data. Within some categories (e.g., Medicines at 46% coverage, Cakes & Pastries at 79%), the subset with unit data may not be representative of the full category. The main chained index does not have this limitation.
  • Pack size parsing. Some display_unit strings cannot be parsed (unusual formats, bundled items, ambiguous units). These products are excluded from unit price calculations.
  • Short time series. With only a few days of data, the index is sensitive to daily promotions and stock fluctuations. Trends become more reliable as more data accumulates.
  • Not official statistics. This tracker is a personal project for monitoring grocery inflation and is not affiliated with SingStat, any supermarket chain, or any government body.

8. Technical Architecture

The system consists of Python scrapers running on a daily cron schedule, storing data in a SQLite database. The dashboard is a static HTML file regenerated after each scrape, with all data embedded as JSON — no backend server required.
